Getting Started

Open the app, load a Jira CSV export, then choose a report and narrow the dataset with filters.

The app works entirely in the browser. Your CSV is parsed locally, and exports are generated locally too.

What CSVs work best

The app is built for Jira CSV exports and recognizes common fields such as Issue key, Summary, Issue Type, Status, Status Category, Parent key, Sprint, Assignee, and the story-point fields.

Duplicate headers

If Jira exports duplicate column names, the app keeps them and disambiguates them internally so the file can still be analyzed.

Missing fields

The CSV can still load if some columns are missing, but some filters or reports may be less useful when those fields are absent.

Estimates And Mismatch

The app reads up to three estimate fields: Story Points, Dev Story points, and Test Story Points.

For totals and summary views, it chooses one effective estimate in this order: Story Points first, then Dev Story points, then Test Story Points.

What "Estimate mismatch" means

The issue is flagged when at least two of those numeric estimate fields are present and they do not match each other. Example: Story Points = 5 and Dev Story points = 8. If only one field has a value, or all populated values are the same, it is not a mismatch.

Missing Story Points, Missing dev points, and Missing test points are separate checks. A row can have an effective estimate and still be missing one of the other point fields.

Filters

All filters combine together. In other words, the app keeps only issues that satisfy every selected dropdown, search term, preset, and checkbox.

Report And Search
  • Report changes the active output view.
  • Quick Preset applies a saved logic bundle before the other filters.
  • Search matches text inside issue key, summary, parent key, parent summary, and assignee.
Core Dropdown Filters
  • Issue Type filters to one Jira issue type value.
  • Status filters by the exact status text from the CSV.
  • Status Category filters by the Jira category, usually To Do, In Progress, or Done.
  • Parent Key filters to one parent key. (None) means the issue has no parent key at all.
  • Parent Kind groups parent relationships into Epic, Feature, Other Matched Parent, Unmatched Parent Key, and No Parent Key.
  • Assignee filters by assignee name. Empty assignees are normalized to Unassigned.
  • Sprint filters to one sprint value. (None) means the issue has no sprint entries.

Quick Presets

None

Does not pre-filter anything.

Active Delivery

Keeps all issues that are not in the Done status bucket.

Planning Gaps

Keeps active issues that are missing story points, dev points, test points, description, or a valid parent relationship.

Needs Triage

Keeps active issues that are unassigned, critical/blocker, missing a matched parent, or missing a description.

Ready For Grooming

Keeps active non-container issues in the To Do bucket that are missing story points, missing a description, or missing a valid parent relationship.

Reports

Filtered Issues

The raw issue-level view after your current filters. This is the best report for ad hoc inspection and export.

Story To Parent Mapping

Shows each issue with its parent key, whether that parent was found in the loaded dataset, the matched parent type, and the issue's parent kind classification.

Missing Story Points

Shows non-container issues where the Story Points field is blank. This report does not count epics, features, or initiatives.

Missing Parent

Shows work items such as stories, bugs, tasks, and subtasks that have no parent key or whose parent key does not match any issue in the loaded CSV.

Sprint Summary

Aggregates the filtered issues by sprint and shows issue count, total effective estimate, status buckets, and unestimated count. Issues with no sprint are grouped under (No Sprint).

Assignee Workload

Aggregates the filtered issues by assignee with counts, total effective estimate, status buckets, and unestimated count.

Epic Breakdown

Shows container issues with their child issues underneath. Child issues whose parent does not resolve to a loaded container are grouped into (No Epic).

Exports

Export Report

Downloads the currently visible report, using the current filters and the currently selected report layout.

Export Cleaned CSV

Downloads the original dataset with fully empty columns removed. It does not export just the filtered rows, and it does not rewrite issue values. It is mainly for cleaning wide Jira exports.

Reset Filters

Returns all filters to their default state and switches the active report back to Filtered Issues.

Helpful Notes

Parent matching only works inside the loaded CSV. If a child points to a parent issue that is not included in the export, that relationship shows up as Unmatched Parent Key.

Status bucket logic is simplified. The app maps Jira Status Category values into three buckets for summary purposes: todo, in_progress, and done.

Epic Breakdown uses container logic. In this app, Epic, Feature, and Initiative are treated as parent containers.

Everything is static-site friendly. Because the app and this guide are plain static files, both pages work cleanly on GitHub Pages.
